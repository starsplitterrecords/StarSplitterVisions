# Star Splitter Visions

Minimal Vite + React homepage project.

## Run locally

```bash
npm install
npm run dev
```

## Content files

Static content is served from `public/content`:

- `public/content/series.json`
- `public/content/releases.json`
- `public/content/pages.json`

## Publishing Workflow

The publishing workflow is intentionally Codex-assisted.

The admin area does **not** publish directly to GitHub.

The admin area is a local content-preparation and package-generation tool. It helps normalize content metadata, image destinations, JSON entries, and publishing instructions into a Codex-ready package.

Codex then applies that package to the repository by adding image files, updating JSON data files, running validation/build checks, and preparing a pull request.

1. Prepare content in the admin helper.
2. Select or reference image files locally.
3. Generate a Codex-ready publishing package.
4. Paste the package into Codex.
5. Codex adds assets under `/public/images/...`.
6. Codex updates the relevant JSON files.
7. Codex runs validation and build checks.
8. Review the Vercel preview.
9. Merge the PR when the preview is correct.

This keeps GitHub credentials, repository mutation, pull requests, and deployment outside the app itself. The admin helper remains a safe package generator instead of becoming a publishing backend.

## Image Path Rules

Image files belong in the repository under:

`public/images/{seriesSlug}/{releaseSlug}/`

Example repo path:

`public/images/vikings-2026/vikings-2026-issue-1/page-001.jpg`

JSON paths must omit `/public`.

Correct JSON path:

`/images/vikings-2026/vikings-2026-issue-1/page-001.jpg`

Incorrect JSON path:

`/public/images/vikings-2026/vikings-2026-issue-1/page-001.jpg`

Cover image example:

- Repo path: `public/images/{seriesSlug}/{releaseSlug}/cover.jpg`
- JSON path: `/images/{seriesSlug}/{releaseSlug}/cover.jpg`

## Files Codex May Update

Depending on the package, Codex may update:

- `public/content/series.json`
- `public/content/releases.json`
- `public/content/pages.json`

Codex may also update fields that control release/page ordering, image references, and optional extras/soundtrack metadata that are stored in those content files.

## Codex Content Publishing Package (Standard Shape)

````md
# Codex Content Publishing Package

## Task

Add or update release `{releaseSlug}` for series `{seriesSlug}`.

## Intent

This package was generated by the admin helper. Apply it to the repository as a content/data update. Do not build new publishing infrastructure.

## Files to Add

Place the following image files in the repo:

- Source file: `cover.jpg`
  Target repo path: `public/images/{seriesSlug}/{releaseSlug}/cover.jpg`
  JSON path: `/images/{seriesSlug}/{releaseSlug}/cover.jpg`

- Source file: `page-001.jpg`
  Target repo path: `public/images/{seriesSlug}/{releaseSlug}/page-001.jpg`
  JSON path: `/images/{seriesSlug}/{releaseSlug}/page-001.jpg`

## JSON Updates

Update the relevant content JSON files.

### Series (`public/content/series.json`)

```json
{
  "slug": "{seriesSlug}",
  "title": "{seriesTitle}"
}
```

### Release (`public/content/releases.json`)

```json
{
  "id": "{releaseSlug}",
  "seriesSlug": "{seriesSlug}",
  "title": "{releaseTitle}",
  "coverImage": "/images/{seriesSlug}/{releaseSlug}/cover.jpg"
}
```

### Pages (`public/content/pages.json`)

```json
[
  {
    "seriesSlug": "{seriesSlug}",
    "releaseSlug": "{releaseSlug}",
    "pageNumber": 1,
    "image": "/images/{seriesSlug}/{releaseSlug}/page-001.jpg"
  }
]
```

## Validation Checklist

After applying this package:

- Confirm all referenced image files exist.
- Confirm no JSON image path starts with `/public`.
- Confirm release slug matches page release slug.
- Confirm series slug exists.
- Confirm page order is correct.
- Run project build.
- Open the Vercel preview and visually inspect the release.
````

## Admin Publishing Checklist

Before generating the Codex package:

- Confirm the series slug is correct.
- Confirm the release slug (`releases[].id`) is final.
- Confirm the release title is final.
- Confirm cover artwork is selected.
- Confirm all page images are in the correct order.
- Confirm page numbering starts where expected.
- Confirm all external links are valid.
- Confirm all JSON image paths use `/images/...`, not `/public/images/...`.

Before merging the PR:

- Confirm the build passes.
- Confirm the Vercel preview loads.
- Confirm the series page displays correctly.
- Confirm the release page displays correctly.
- Confirm the reader page displays all pages in order.
- Confirm cover art renders correctly.
- Confirm no broken images or undefined text appears.

## Common Publishing Mistakes

Avoid these mistakes:

- Do not put `/public` in JSON image paths.
- Do not create a release without a matching series slug.
- Do not create pages with a release slug that does not exist.
- Do not rename image files after generating the package unless JSON paths are updated too.
- Do not treat the admin helper as a live CMS.
- Do not add GitHub tokens or publishing credentials to the frontend.
- Do not merge without checking the preview.

## Codex Handoff Rules

When applying a generated publishing package, Codex should:

- Preserve the existing schema.
- Use existing data conventions.
- Add only the requested assets and JSON entries.
- Avoid large unrelated refactors.
- Avoid changing routing unless the package explicitly requires it.
- Keep generated image references stable.
- Run build/validation checks before completing the task.
- Report any schema mismatch instead of guessing silently.

## Content Schema

### 1) Series schema (`public/content/series.json`)

Each object in `series[]` should include:

- `slug` (string)
- `title` (string)
- `logoText` (string)
- `status` (string)
- `format` (string)
- `genre` (string[])
- `tone` (string[])
- `tagline` (string)
- `shortDescription` (string)
- `longDescription` (string)
- `audiencePromise` (string)
- `coreConflict` (string)
- `worldPremise` (string)
- `seriesEngine` (string)
- `accentColor` (hex color string)
- `secondaryColor` (hex color string)
- `backgroundTone` (string)
- `heroImage` (web path)
- `coverImage` (web path)
- `thumbnailImage` (web path)

Allowed `status` values:

- `ongoing`
- `in development`
- `complete`

### 2) Release schema (`public/content/releases.json`)

Each object in `releases[]` should include:

- `id` (string)
- `seriesSlug` (string)
- `title` (string)
- `issueNumber` (number)
- `description` (string)
- `releaseDate` (`YYYY-MM-DD`)
- `status` (string)
- `image` (web path)
- `coverImage` (web path)
- `heroImage` (web path)
- `ctaLabel` (string)

Allowed `status` values:

- `draft`
- `scheduled`
- `published`

Notes:

- `id` is the route slug used by `/releases/{id}` and `/read/{id}`.
- Public UI must not show `draft` releases.
- `scheduled` releases are hidden until `releaseDate` is in the past or present.
- `scheduled` releases should always include a `releaseDate`.

Status visibility checks:

- `draft`: hidden from homepage, series archives, search, release routes, and reader routes.
- `scheduled`: hidden while `releaseDate` is in the future; visible once `releaseDate` is reached.
- `published`: visible immediately.

### 3) Page schema (`public/content/pages.json`)

Each object in `pages[]` should include:

- `seriesSlug` (string)
- `releaseSlug` (string)
- `pageNumber` (number)
- `releaseDate` (`YYYY-MM-DD`)
- `title` (string)
- `caption` (string)
- `image` (web path)

Rules:

- `releaseSlug` must match an existing `releases[].id`.
- `seriesSlug` must match the same release object's `seriesSlug`.
- `pageNumber` should be numeric.
- Image path convention: `/images/{seriesSlug}/{releaseSlug}/page-001.jpg`.

## Relationship map

- `series.slug` ← canonical series identifier.
- `releases.seriesSlug` → must match `series.slug`.
- `releases.id` ← canonical release identifier and route slug.
- `pages.releaseSlug` → must match `releases.id`.
- `pages.seriesSlug` → must match the parent release's `seriesSlug`.
